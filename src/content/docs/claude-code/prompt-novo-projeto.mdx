Cole o bloco abaixo no Claude Code **dentro do diretório do novo projeto**. O Claude vai explorar o projeto, criar os arquivos adaptados e guiar você pelo que não conseguir inferir sozinho.

<Aside type="tip">
  O prompt instrui o Claude a ler o projeto antes de criar qualquer coisa. Quanto mais o projeto já tiver (package.json, configs, código), mais preciso será o resultado.
</Aside>

---

`````text
Vou configurar este projeto para trabalhar com Claude Code. Antes de criar qualquer arquivo, explore o projeto:

1. Leia package.json (ou equivalente: requirements.txt, go.mod, Cargo.toml, etc.)
2. Liste a estrutura de pastas (apenas primeiro e segundo nível)
3. Verifique se já existe CLAUDE.md, .claude/, .eslintrc, prettier.config, tsconfig, etc.
4. Identifique: linguagem, framework, gerenciador de pacotes, scripts disponíveis, banco de dados (se houver)

Com base no que encontrar, crie os arquivos abaixo **já adaptados ao projeto**. Não use placeholders genéricos — preencha com o que descobriu. Para o que não conseguir inferir, marque com ← PERGUNTAR e me pergunte depois.

---

ARQUIVO 1: CLAUDE.md (na raiz do projeto)

Regras:
- Máximo 200 linhas. Seja conciso.
- Só escreva o que o Claude precisa saber para trabalhar neste projeto.
- Não repita o que já está em configs (eslint, prettier, tsconfig).
- Git Workflow deve ser resumido (~15 linhas max). O detalhado vai no WORKFLOW.md.

Estrutura obrigatória:

# [Nome do Projeto]
[Uma linha sobre o que é]

## Comandos
[Preencher com os scripts reais encontrados no projeto]

## Stack e Arquitetura
[Framework, banco, estrutura de pastas dos módulos principais]

## Convenções
[Padrões de código específicos deste projeto: shape de retorno de API, módulo de logger, validação, etc.]
[Se não houver convenções claras ainda, sugerir 3-4 baseadas no stack detectado e marcar ← PERGUNTAR]

## Atenção
[Armadilhas não-óbvias: banco real em testes, strict mode, variáveis de ambiente necessárias, etc.]

## Git Workflow (resumido)

Antes de qualquer trabalho: git fetch origin && git status

Branches a partir de main. Nunca commitar direto em main.
Convenção: feat/ISSUE-X-descricao, fix/ISSUE-X-descricao, refactor/, test/, chore/
← PERGUNTAR: prefixo de issue (ex: FIL-, APP-) e issue tracker usado (Linear, Jira, GitHub Issues)

Commits: Conventional Commits com Closes ISSUE-X no footer.
PRs: via gh CLI com corpo contendo Closes ISSUE-X.
Merge: squash + delete branch.

## O que NUNCA fazer

Git:
- Nunca commitar direto em main — sempre branch + PR

[Adicionar regras de UI/Design se for projeto frontend — ← PERGUNTAR]
[Adicionar regras de banco se usar banco — baseado no que encontrou]

## Fluxo Completo de uma Feature

## Política de Merge ← PERGUNTAR
Requer revisão antes do merge:

- [sugerir categorias simples]

Adaptar permissões ao stack real do projeto:

APÓS CRIAR OS 3 ARQUIVOS:

1. Liste todas as seções marcadas com ← PERGUNTAR
2. Pergunte sobre cada uma em sequência
3. Edite os arquivos conforme minhas respostas
4. No final, mostre um resumo do que foi configurado e sugira próximos passos:
   - Criar CLAUDE.local.md para preferências pessoais (gitignored automaticamente)
   - Criar .claude/rules/ quando o CLAUDE.md crescer demais
   - Criar .claude/commands/ para workflows frequentes (ex: review.md → /project:review)
5. Commite tudo: chore: adiciona CLAUDE.md, WORKFLOW.md e .claude/settings.json
`````

---

## Como funciona

1. **O Claude explora o projeto** — lê package.json, configs e estrutura de pastas antes de criar qualquer coisa
2. **Gera arquivos já adaptados** — comandos reais, stack real, paths reais. Sem `[Ex: ...]` genéricos
3. **Pergunta só o que não consegue inferir** — prefixo de issue, tracker usado, convenções específicas do time
4. **Sugere próximos passos** — CLAUDE.local.md, rules/, commands/ para quando o projeto crescer

<Aside type="note">
  O CLAUDE.md fica enxuto (~100-150 linhas) porque o Git Workflow é resumido — o detalhamento completo vai no WORKFLOW.md, que é para humanos consultarem.
</Aside>
